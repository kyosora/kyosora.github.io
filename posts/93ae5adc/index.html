SQL Server CSV 匯出實戰：SQLCMD、BCP、SSIS 怎麼選怎麼調

發表於2024-12-17|資料庫
|總字數:3.5k|閱讀時間:12分鐘

老闆丟一句「這個報表，越快越好」，你打開資料表一看——好幾千萬筆。這時候用對工具跟用錯工具，差的不是幾分鐘，是幾個小時。

SQL Server 匯出 CSV 最常用的三個工具是 SQLCMD、BCP、SSIS。網路上很多範例其實是把它們的角色搞混的：最典型的就是把 bcp 命令當成 T-SQL 字串丟進 sp_executesql，抄回去一跑就語法錯誤。這篇把每個工具的定位講清楚，範例全部以可實際執行為前提，並附上分批匯出大型資料的正確寫法。

三個工具，三種角色

先把定位搞對，後面才不會誤用。

BCP（Bulk Copy Program）是獨立的命令列工具，走大量複製 API，匯出大表速度最快、記憶體佔用低。它不是 T-SQL，是在作業系統的命令提示字元（或透過 xp_cmdshell）執行的 .exe。

SQLCMD 也是命令列工具，主要拿來跑 .sql 指令稿、做自動化排程。它本身不是高效能匯出引擎——真要快速匯出大量資料，多半還是在 SQLCMD 指令稿裡呼叫 BCP。

SSIS（SQL Server Integration Services）是視覺化 ETL 平台，強在「匯出過程中要做轉換」：欄位對應、編碼轉換、異常過濾、條件分流。代價是設定與維護成本高。

工具 角色 強項 弱項
BCP 命令列匯出引擎 大量資料最快、資源佔用低 彈性低、特殊處理麻煩
SQLCMD 指令稿 / 排程外殼 自動化、整合方便 不是匯出引擎，常需搭 BCP
SSIS 視覺化 ETL 複雜轉換、流程整合 設定複雜、維護成本高

效能上，純匯出大表 BCP 通常領先，SSIS 因為多了緩衝與轉換管線，純匯出反而比 BCP 慢；但一旦匯出途中需要做轉換，SSIS 一次到位的價值就出來了。實際數字會被磁碟 I/O、網路頻寬、資料表結構、索引拖很大，建議在自己環境跑小樣本實測，不要照搬別人的秒數。

BCP 實戰

BCP 是這三者裡最該先學會的，因為大型匯出最後幾乎都會落到它身上。

關鍵旗標：

  • queryout：表示來源是一段查詢（如果是整張表用 out）。

  • -t","：欄位分隔符號設為逗號。

  • -T：使用 Windows 整合驗證；要用 SQL 帳密就改成 -U 帳號 -P 密碼。

-T 跟 -U/-P 擇一，不要同時給。

效能相關旗標

  • -b 10000：每批次列數，影響交易與記憶體配置。
  • -a 16384：網路封包大小（位元組），跨網段匯出時調大有機會減少往返。

這裡要點名一個網路上常見的幻覺旗標：**bcp 沒有 -O 這個選項**。保留識別欄位（identity）值的旗標是 -E，而且 -E 只在「匯入（in）」時才有意義——把資料檔裡的 identity 值原樣寫回去；對 queryout 匯出根本用不到。如果你看到範例寫 -O -- 保持身份值，那是錯的，照抄會直接報錯。至於 -h "TABLOCK" 這類載入提示，同樣是匯入時的最佳化，匯出時不適用。

幾千萬筆一次匯出，記憶體與失敗重試都是麻煩，分批比較穩。重點是分批的邊界要組進實際送給 bcp 的命令字串裡——因為 bcp 啟動的是獨立的新連線，看不到你在外層 T-SQL 宣告的變數。換句話說，你不能寫 WHERE ID BETWEEN @StartID AND @EndID 然後期待 bcp 認得 @StartID；得在組命令時就把數值字串化拼進去。

bcp 是 SQL Server 外部的作業系統行程，它在引擎內只會以一般的 SELECT 連線出現，command 欄位永遠不會是 'BCP'，這個計數永遠回 0，節流形同虛設。而且 xp_cmdshell 是同步阻塞的——一次只會等一個 bcp 跑完才回來，根本不會有多個並行。真要平行，做法是在作業系統層各自啟動多個 bcp 行程（例如多個排程工作、批次檔背景啟動、或 PowerShell 的 Start-Job），讓 OS 去調度，而不是在 T-SQL 裡自己數。

SQLCMD 實戰

SQLCMD 的價值在指令稿化與排程，不在匯出效能本身。

指令稿變數

要記得 :setvar 只是宣告 sqlcmd 指令稿層級的變數，本身不會對 SQL Server 做任何設定。它能在後續用 $(OutputPath) 這種 $() 語法被展開——但這套展開只在 sqlcmd 環境裡有效。把 $(StartID) 寫進一段準備交給 bcp 的字串、又不是在 sqlcmd 跑，它不會展開，會原樣留著變成壞檔名。

實際匯出仍是在指令稿裡呼叫 bcp（透過命令列或 xp_cmdshell），SQLCMD 負責把流程串起來、接排程。

別把 -m 當記憶體參數

網路上反覆出現一個錯誤，要特別澄清：

sqlcmd -m 是「錯誤訊息嚴重性等級門檻」（error level），只有嚴重性大於等於這個值的訊息才會輸出。它的有效範圍是 -1 到 25，寫 65535 直接超出範圍。-m -1 代表連資訊性訊息都印出來。它跟記憶體一點關係都沒有——SQL Server 的記憶體上限是用 sp_configure 'max server memory' 設定的，命令列工具根本沒有控制伺服器記憶體的旗標。

-b 才是有用的：遇到錯誤時讓 sqlcmd 以非零代碼結束批次，方便排程判斷成敗。

SSIS 實戰

SSIS 適合匯出途中要轉換的場景。它的設定主要在圖形介面拉元件，這裡只點幾個容易抄錯的地方。

緩衝區屬性屬於資料流程

效能調校常碰到這幾個屬性，但它們是資料流程工作（Data Flow Task）的屬性，在設計工具的屬性視窗設定，不是隨便在某段腳本裡賦值的變數：

  • DefaultBufferMaxRows：每個緩衝區的最大列數（預設 10000）。
  • DefaultBufferSize：每個緩衝區的位元組大小（預設約 10 MB）。
  • BufferTempStoragePath：緩衝溢出時的暫存路徑。

編碼：UTF-8 沒那麼單純

匯出 UTF-8 CSV 常見寫法是 -C 65001，但有幾個坑要先知道：

  • UTF-8 的 code page 支援從 SQL Server 2014 SP2 才加入，更舊的版本沒有。
  • UTF-8 是 8 位元編碼，要搭 -c（字元格式）使用，不要用 -w（-w 是 UTF-16 寬字元）。
  • bcp 輸出的 UTF-8 檔不含 BOM，某些下游程式（特別是 Excel 直接開）可能因此判錯編碼把中文變亂碼。

所以單寫 -C 65001 不保證拿到下游都吃得乾淨的 UTF-8。版本不夠新、或下游要 BOM 的話，得另外處理。不確定環境支援度時，先在小樣本上驗證再放上生產。

怎麼選

把工具對到任務上，大致是這樣：

  • 純匯出、量大：BCP。簡單結構、追求速度，沒對手。
  • 要排程、要串流程：SQLCMD 當外殼，內部呼叫 BCP。
  • 匯出途中要轉換：SSIS。欄位對應、編碼轉換、異常過濾一次到位。
  • 大量資料 + 複雜轉換：BCP 先快速倒出原始檔，再用 SSIS 做轉換，分兩階段比硬塞單一工具穩。

收尾

三個工具裡，BCP 值得最先投資——大型匯出最後幾乎都繞回它。而最該避開的不是哪個工具不好，是把它們的角色搞混：bcp 是命令列工具不是 T-SQL，要從資料庫裡觸發就用 xp_cmdshell；分批的邊界得拼進命令字串，因為 bcp 看不到你的 T-SQL 變數；sqlcmd -m 是錯誤等級不是記憶體。把這幾條釐清，你寫出來的匯出腳本才是能真的跑起來的腳本，而不是看起來很專業、一執行就報錯的範本。
