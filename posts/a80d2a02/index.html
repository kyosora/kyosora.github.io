密碼或大檔不小心推上 Git？用 git-filter-repo 安全清掉歷史

發表於2024-11-21|工具與生產力
|總字數:901|閱讀時間:3分鐘

把密碼、API 金鑰或一個幾百 MB 的檔案不小心 commit 進 Git、又推上遠端——這時候光 git rm 沒用，它還躺在歷史裡，任何人 clone 下來都翻得到。要真正清掉，得重寫 Git 歷史。

Git 內建的 filter-branch 能做，但它慢、容易出錯，官方文件自己都標示不建議用了。現在的標準工具是 git-filter-repo。

什麼是 git-filter-repo

用 Python 寫的開源工具，專門重寫 Git 歷史，是 filter-branch 的官方推薦替代品。比起 filter-branch，它快上一個量級、命令更直覺，還會自動做安全檢查（例如要求你在乾淨的 clone 上操作）。

安裝

情境一：移除意外推送的敏感資料

--invert-paths 是「保留以外的全部」，配合 --path 就是把這些路徑從整個歷史刪掉。

但有件事要先講清楚：把檔案從歷史移除，不等於敏感資料安全了。 只要它曾經推上遠端，就要當作已經外洩——

  • 立刻撤銷並更換所有外洩的密碼、金鑰
  • 檢查其他分支、fork、CI 日誌是否也有
  • 之後用 .gitignore + pre-commit hook（如 git-secrets）擋住下一次

清歷史是止血，換金鑰才是真正解決。

情境四：把子目錄變成根目錄

常用在從 monorepo 拆出一個獨立專案，連歷史一起搬走。

用之前要知道的幾件事

  1. 在複製出來的 repo 操作，原始 repo 留著當保險。
  2. 重寫不可逆，確定真的要做。
  3. 之後要 git push --force 覆蓋遠端，先跟團隊講好，不然會把別人的本地搞亂。
  4. 不用手動下 git gc。git-filter-repo 跑完會自己 expire reflog、repack 清掉舊物件。

整個 git-filter-repo 的世界都是位元組——路徑、email、訊息、檔案內容全是 bytes，這點一致。

跟 filter-branch 比

特性 git-filter-repo git filter-branch
速度 快 慢
上手難度 簡單 複雜
安全檢查 有 無
官方態度 推薦 不建議使用

收尾

git-filter-repo 把「清掉 Git 歷史裡的東西」變得又快又相對安全。真正要記住的就兩點：callback 全用 bytes（加 b）、清完敏感資料一定要換金鑰。其他照情境套就行。
