六、效能最佳化建議

索引寫法決定能不能走索引。 STIntersects()、STContains()、STWithin() 能觸發空間索引；STDistance() 直接放 WHERE 子句則不行——必須先用 STBuffer 建緩衝區再呼叫 STIntersects，做完 bounding-box 過濾後才在外層用 STDistance 精確篩選。

非空間條件先過濾。 WHERE Type = '住宅' AND point_geometry.STIntersects(@zone) = 1 比反過來快得多；SQL Server 會先走一般 B-tree 索引縮小候選集，再跑空間索引。

自我 JOIN 的代價是 O(n²)。 10,000 筆點位的全配對距離是 1 億次計算。若只需「每個點找 1000 單位內的鄰居」，應先用 STBuffer 限制半徑，不要無條件 CROSS JOIN 再用 STDistance 過濾。

UnionAggregate 在 CROSS APPLY 裡代價驚人。 對每一列外表都重新聚合一次全表——應在 CTE 或子查詢中預先算好一次，再拿結果做 STContains。

SRID 不一致會導致靜默錯誤，不是執行期例外。 STDistance、STIntersects 在 SRID 不符時回傳 NULL 或 0，查詢不會報錯但結果完全錯誤；建立資料時就應確認 SRID，不要等出問題再查。

結語

空間查詢裡最常踩的坑集中在三個地方：索引寫法（STDistance 放 WHERE 就繞過索引）、型別混用（geometry 的距離單位不是公尺）、以及 O(n²) 的自我 JOIN。搞清楚這三點，其他的查詢組合方式都能照 SQL Server 的一般調校邏輯處理。
