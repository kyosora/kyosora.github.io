我把重複貼了一百次的 prompt，變成一個 AI 會自己想起來用的技能

發表於2026-06-02|AI 輔助開發開發流程
|總字數:2.4k|閱讀時間:7分鐘

我有一段 prompt，大概貼了一百次。

每次要 AI 幫我寫部落格，我就把那段「語氣要這樣、禁用這些詞、結構照這個走、寫完先給我審不要直接發」的長指令翻出來貼一遍。貼到後來我自己都煩了——更糟的是每次貼的版本還會微妙地漂移，這次多一句、下次漏一句，輸出品質跟著飄。

後來我把它封裝成一個 skill。現在我只要說「幫我寫篇 blog」，那整套規則 AI 會自己想起來、自己套上。這篇講的就是這件事：什麼東西該封裝成可觸發的技能、怎麼封裝、以及我封裝壞掉的那幾次。

先分清楚你手上是哪種東西

「把常用的東西存起來」有好幾個層次，差別不在存不存，在存的是什麼，以及它「怎麼被叫出來」。

形式 存的是什麼 怎麼被叫出來
別名 / 片段 一行固定指令、固定文字 你手動打
prompt 範本 一段可填空的長 prompt 你手動貼
指令（command） 一個明確的 /xxx 工作流 你手動唸出那個指令
技能（skill） 程序 + 規則 + 觸發時機 AI 在對的情境自己接上

最底層的別名（alias），gco 等於 git checkout，只幫你少打字，不含任何判斷——你得自己知道什麼時候用、用完接什麼。

往上是 prompt 範本，存了內容，但「什麼時候用」「用完接什麼」還是靠你的腦袋，而且要你主動想起它存在。

再往上是 command。在 Claude Code 裡就是那些 / 開頭的東西，封裝了一整套流程，但你還是得記得它、手動唸出來。

最上層才是 skill。它跟 command 最大的差別在那一欄：skill 能在對的時機自己被叫出來，不需要你記得它存在。我那段貼了一百次的 blog prompt，原本卡在「範本」這層——內容有了，但每次都要我自己想起來、手動貼。把它升到 skill，AI 在我講到「寫文章」時就自動把整套規則接上，我不用再記、不用再貼。

一個 skill 長什麼樣

以 Claude Code 為例，一個 skill 是一個資料夾，入口是裡面的 SKILL.md。放在 ~/.claude/skills/<skill-name>/ 是你個人全域到處能用，放在專案根目錄的 .claude/skills/<skill-name>/ 則是跟著專案走、team 共用。SKILL.md 開頭一段 frontmatter，下面是內容：

name 是這個 skill 的識別名稱，通常跟資料夾同名。真正決定「什麼時候會被想起來」的是 description。下面那段內容，則是被想起來之後 AI 要照著走的 SOP。

平常 AI 的記憶裡，只掛著每個 skill 的 name 和 description 這一行簡介，像一份目錄；主體那幾百行 SOP 並不載入。只有當你當下要做的事跟某條 description 對上了，那一份的完整內容才被讀進來。這叫漸進式揭露（progressive disclosure）。你可以養一百個 skill，平常它們的主體全部沉睡，只有目錄掛在那；需要哪個，哪個的內文才醒——而不是一百份 SOP 同時擠在 context 裡互相干擾。

也正因為平常只有那行 description 代表它，這行字寫得好不好，幾乎決定了整個 skill 有沒有用。

我封裝壞掉的那幾次

封裝 skill 最難的不是寫內容，是寫那行 description。我前幾個 skill 都死在這裡。

第一次，我把 description 寫得太貪心：「處理檔案、文字、資料相關任務時使用」。結果它幾乎每個對話都跳出來——我只是想讀一個設定檔，它也以為輪到它上場。description 涵蓋太廣，等於沒有觸發條件，反而變成雜訊。

矯枉過正的第二次，我把它收得死緊：「當使用者說『請啟動部落格寫作技能』時使用」。這下它乖乖躺著不亂跳了，但問題是——我每次還是得明確唸出那句咒語它才動。那我封裝它幹嘛？繞了一圈又回到「必須記得它存在」。

第三次才抓到手感：description 要描述的是使用者的意圖和情境，用他真的會講的話，不是內部術語。「當使用者要寫 blog、發部落格、寫技術文章時」——這裡面塞了好幾個同義的講法，因為我自己有時說 blog、有時說部落格、有時說「幫我寫篇文章」。把這些自然說法都列進去，命中率才高。

我後來的判準是這樣：

description 寫給「未來那個不記得這個 skill 存在的我」看。他會用什麼話描述他想幹的事，你就把那些話寫進去。

寫完 description，記得測

description 不是寫完就算了，它值得像程式一樣測一遍。

我的土法煉鋼：寫之前先列兩組句子。一組是 8 到 10 個「我希望它觸發」的講法——blog、部落格、幫我寫文章、發一篇、寫個技術紀錄……盡量涵蓋我真實會講的各種說法。另一組是幾個「我絕對不希望它觸發」的場景——我只是想讀一個 markdown 檔、我在問 Markdown 語法、我提到「文章」但其實在講別人的文章。

然後拿這兩組去實際跑跑看：該觸發的有沒有漏掉（漏觸），不該觸發的有沒有亂跳（誤觸）。漏觸就把那個講法補進 description，誤觸就把範圍收窄一點。跑個兩三輪，那行字就調得差不多了。比起寫的時候憑感覺，這個來回幾分鐘的測試，省掉的是之後幾個月「它怎麼又跳出來」或「它怎麼又不動」的煩躁。

那到底什麼該封裝、什麼不該

踩完幾輪，我大致分得出來了。

值得封裝成 skill 的，是那種「多步驟、有固定順序、你會忘記細節」的流程。我的 blog 寫作有十幾條語氣規則加一套發佈流程，每次手動回想一定漏；封裝起來，一條都不會少。debug 的標準排查步驟、提交程式碼前的檢查清單、某個冷門工具的操作 SOP——這些共通點都是：步驟固定、但細節多到記不住。

不值得封裝的，是兩種。一種是單純少打字的東西，那是別名的工作，殺雞用不到牛刀。另一種更要小心：需要你當場判斷、不該被寫死的決策。我不會把「這個架構該怎麼設計」封裝成 skill，因為每個案子的取捨都不一樣，把它變成固定 SOP 等於提前替未來的自己做了不該做的決定。skill 適合封裝「怎麼做」的流程，不適合封裝「要不要做」的判斷。

還有一個常見的失手：把 skill 寫成百科全書。我見過有人把整本框架文件塞進一個 skill，想說「這樣 AI 什麼都查得到」。但 skill 該回答的是「這件事照什麼步驟做」，不是「這個框架的所有 API」。內容越長，每次觸發吃掉的 context 越多，漸進式揭露省下來的好處就被你自己吐回去了。真要放大量參考資料，Claude Code 允許你在 skill 資料夾裡開附檔（references 之類），主檔只留 SOP、需要時才去翻附檔，別全堆在 SKILL.md 裡。

收一下

從「貼了一百次的 prompt」到「一個會自己跳出來的 skill」，這條路我走得有點繞，但回頭看就三件事：

  • 分清層次——別名幫你少打字，範本幫你存內容，command 幫你存流程但要手動叫，skill 連「什麼時候用」一起封裝、自己接上
  • description 是靈魂——用未來的你會講的自然話去寫，太泛會吵、太窄沒用，寫完拿正反兩組句子測一測
  • 封裝流程，不封裝判斷——固定步驟適合 skill，當場取捨的決策留給人

這套思路能不能搬到別的平台，得看你分清楚搬的是哪一層。「把重複、固定、會忘的流程沉澱成一個模組」這個封裝思路，在 ChatGPT 的自訂指令、各家的 agent 設定上都通用。但「靠 description 自動觸發、主體延後載入」這個機制，是 Claude Code 這類 Agent Skills 才有的；有些平台只能做到常駐指令（永遠在線、不會休眠），有些甚至只是手動範本。能搬的是封裝的思路，不一定是自動觸發的行為——別把後者當成理所當然。

那段 blog prompt 我現在再也沒貼過了。AI 比我還記得那些規則——這大概是我今年做過投報率最高的一次偷懶。
