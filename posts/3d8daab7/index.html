使用 SQL Server Resource Governor 限制特定資料庫的資源使用

發表於2024-08-08|資料庫
|總字數:1.5k|閱讀時間:5分鐘

1. 背景

某次正式環境使用者抱怨網頁速度極慢，卡了好幾分鐘，我連上 SSMS 查詢也一樣卡。追查後發現另一個專案的排程在跑，瘋狂吃 CPU 和記憶體，把整台 SQL Server 拖垮。理想解法是切獨立主機，但沒有資源時，Resource Governor 是次佳選項——不需重建環境，直接在現有執行個體限制特定資料庫的耗用量。

版本限制（重要）： Resource Governor 在 SQL Server 2022 及以前僅支援 Enterprise 與 Developer Edition，Standard Edition 不支援。若在 Standard 上執行下列指令會直接報錯，請先用 SELECT SERVERPROPERTY('Edition') 確認版本再繼續。

2. Resource Governor 的核心概念

Resource Governor 管理的單位有三層：

  • 資源池 (Resource Pool)：定義 CPU 與記憶體的用量上限，可建立多個池子分別對應不同優先級的工作負載。
  • 工作負載群組 (Workload Group)：關聯到某個資源池，細分同一個池子內的工作分類。
  • 分類函數 (Classifier Function)：每個連線進來時由 SQL Server 呼叫此函數，回傳應該進入哪個工作負載群組的名稱。

重要：MAX 與 CAP 的差異

資源池有兩種 CPU 上限，語意完全不同，常見混淆：

參數 行為
MAX_CPU_PERCENT 軟性上限：只在多個池子互搶 CPU 時才強制執行。伺服器 CPU 空閒時，該資源池可超過此值，甚至用到 100%。
CAP_CPU_PERCENT 硬性上限：不論伺服器是否空閒，該資源池永遠不能超過此值。

若你的目的是「無論如何不讓這個資料庫超過某個 CPU 比例」，必須用 CAP_CPU_PERCENT。只設 MAX_CPU_PERCENT 在離峰時段根本沒有限制效果。

記憶體的 MAX_MEMORY_PERCENT 同樣是軟性上限，邏輯相同。

6. 常見問題

Q：RECONFIGURE 回報「找不到分類函數」
確認函數存在於 master 資料庫，且函數名稱含結構描述（dbo.fn_WorkloadClassifier）。

Q：RECONFIGURE 回報「分類函數不允許存取使用者資料表」
函數內部呼叫到了存取使用者資料表的程式碼。分類函數只能用 DB_NAME()、SUSER_SNAME() 等系統函數，無法查自訂資料表。

Q：設了 MAX_CPU_PERCENT = 20，但監控看到 CPU 超過 20%
這是預期行為。MAX_CPU_PERCENT 是軟性上限，伺服器 CPU 空閒時不限制。若需要硬限，改用 CAP_CPU_PERCENT。

Q：Standard Edition 能用嗎？
SQL Server 2022 以前不行，只有 Enterprise 與 Developer Edition 支援。
