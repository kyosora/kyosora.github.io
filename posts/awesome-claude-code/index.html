你的 Claude Code 只用到了 20%？這份 25k Stars 的資源清單讓它變 5 倍強

發表於2026-02-25|AI開發工具
|總字數:1.7k|閱讀時間:5分鐘|瀏覽量:

剛開始用 Claude Code 的時候，我以為它就是個進階版的「對話框」——貼代碼進去，等它回答。

直到我看到 awesome-claude-code 這個 repo，才發現自己根本沒在用這個工具。

這份由社群維護的清單目前有 25k Stars、1.5k Forks，收錄了超過 200 個工具、工作流程、配置模板和自動化腳本。不是那種「整理一堆連結就叫 awesome」的水貨清單——每個項目都要求有實際可用的功能，不收實驗性的佔坑。

拆開來看，你會發現 Claude Code 早就進化成一個完整的開發生態系了。

大多數人都在手動做的事，這裡早有自動化方案

先說最實際的：Hooks。

Claude Code 支援在特定事件觸發時執行自定義腳本——工具執行前後、Session 結束時都能掛載。清單裡有幾個直接可以用的：

  • CCNotify：每次 Claude Code 完成任務就發桌面通知。你終於不用一直盯著終端機等它跑完了
  • Britfix：自動把美式英語拼法轉成英式（color → colour，analyze → analyse）。在有語言規範要求的專案裡省下大量 review 時間
  • auto-git-commit-hook：每次檔案修改後自動暫存並提交，確保任何 AI 生成的變更都有版本記錄

Hooks 的威力在於「你完全不需要記得去做」。把規範寫進自動化，才是真正可持續的工作流。

Claude.md：AI 的工作說明書

在 Claude Code 的設計裡，CLAUDE.md 是你放在專案根目錄的配置文件，告訴 AI 這個專案的規範、架構、和注意事項。

概念上很簡單，但要寫好其實很難。

清單裡收錄了大量社群實戰的範本，涵蓋：

  • TypeScript/Next.js 專案：組件命名規範、狀態管理原則、測試覆蓋率要求
  • Go 微服務：錯誤處理慣例、interface 設計原則、部署流程說明
  • Python 資料科學：環境管理、Notebook 格式規範、模型版本控制
  • 安全敏感專案（Trail of Bits）：漏洞審計流程、代碼審查清單、合規要求

直接拿這些模板改一改，比從零開始寫要快得多，而且這些都是實際用在生產專案上跑過的版本。

讓 Claude Code 自己跑，你去喝咖啡

這才是清單裡最讓我覺得「原來可以這樣」的部分。

Ralph Wiggum 技術

名字很迷，概念很直白：讓 Claude Code 進入自主循環模式，持續執行直到任務完成，中途遇到問題自己想辦法解決。

適合那種「寫完這個模組的測試」這類有明確完成條件的任務。你給任務，它跑到通過為止。

RIPER Workflow

分成五個階段的結構化開發框架：

  1. Research（研究）：理解問題和現有代碼
  2. Innovate（發想）：提出解決方案
  3. Plan（規劃）：建立具體執行計畫
  4. Execute（執行）：實作
  5. Review（審查）：驗證結果

每個階段都有明確的輸入和輸出定義，防止 AI 在還沒搞清楚問題就開始亂寫代碼。

AB Method

spec-driven 的開發工作流。先讓 Claude 寫出詳細的規格文件（A），確認無誤後再執行實作（B）。適合需求複雜或跨多個文件的功能開發。

這幾個工作流的共同邏輯是：把你的思考過程外化成 AI 可以執行的結構，而不是把所有判斷都丟給 AI 自己猜。

工具篇：你可能不知道這些問題有解

用量追蹤

Claude Code 按 token 計費，但預設介面完全看不出花了多少。這個問題已經有幾個解法：

  • ccflare：視覺化的用量 Dashboard，看 token 消耗、費用趨勢、模型使用分佈
  • Claude Code Usage Monitor：終端機即時顯示目前 Session 的用量，快到上限前警告你

在對帳單出來之前主動掌握成本，比事後驚嚇好多了。

多代理協調

當任務複雜到需要多個 AI 並行處理時：

  • Claude Squad：在一個 workspace 裡管理多個 Claude Code 實例，各自負責不同任務
  • Auto-Claude：自動根據任務類型調度不同的 agent 配置

這類工具的使用場景是：後端 API、前端組件、測試代碼三個部分可以同時進行，最後整合，而不是序列等待。

配置管理

  • ClaudeCTX：一個指令切換整個 Claude Code 的配置環境。在工作專案和個人專案之間切換，不用手動複製貼上不同的 CLAUDE.md
  • claude-rules-doctor：靜態分析你的 .claude/rules/ 目錄，找出路徑 glob 沒有匹配到任何實際文件的死規則。防止你以為規則生效但其實沒有

跟你的 IDE 整合

如果你不喜歡來回切換終端機和編輯器，清單裡有現成的整合方案：

  • VS Code Extension：在 VS Code 側邊欄直接操作 Claude Code，不用離開編輯器
  • Neovim Plugin：claudecode.nvim，支援 buffer 共享和即時代碼操作
  • Emacs Package：claude-code.el，整合進 Emacs 的工作流
  • IntelliJ Plugin：JetBrains 系列 IDE 的整合

Status Line：讓終端機告訴你現在的狀態

這類工具比較小眾，但用過就回不去：

  • ccstatusline：在 shell prompt 顯示目前 Claude Code 的狀態、token 用量、活躍 session 數
  • claude-powerline：Powerline 風格的狀態列，顯示 git 資訊、費用警告、模型版本

對於整天在終端機裡工作的人，這些資訊直接放在 prompt 比去找 Dashboard 方便得多。

怎麼用這份清單

推薦順序：

第一步：先看 CLAUDE.md 範本區，找一個跟你目前專案技術棧最接近的，抄下來改成自己的版本。這是投入產出比最高的動作。

第二步：裝 CCNotify 或類似的通知工具。這個幾分鐘就能設好，立刻改善你等待結果的體驗。

第三步：如果你用 Claude Code 跑比較長的任務，看一下 Ralph Wiggum 或 RIPER 這類工作流，找一個適合你工作模式的套進去。

第四步：根據你的痛點選工具。用量追蹤、多代理協調、IDE 整合——這些不是非用不可的，但遇到對應問題的時候，知道有解才不會覺得沒辦法。

核心要點：

  • 清單本身是活的：每週都有新內容加進來，GitHub Watch 起來
  • 不要想全部都學：選 2-3 個跟你現在工作最相關的，先把那些用熟
  • CLAUDE.md 是最高槓桿的投資：這個文件寫好，等於讓每一次 Claude Code 互動都帶著你的上下文
  • Hook 把規範自動化：不要靠記憶和自律，把你想執行的規範寫進自動化腳本
