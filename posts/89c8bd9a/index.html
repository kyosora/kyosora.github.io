Vue3 SFC Loader 深入解析：在瀏覽器直接載入 Vue 單一檔案元件

引言

在 Vue.js 開發中，單一檔案元件（Single File Components，SFC）允許我們將元件的模板、邏輯和樣式封裝在一個 .vue 檔案裡。大部分情況下 SFC 在建置階段就編譯成 JavaScript，但有時我們需要在執行期動態載入這些檔案——這就是 vue3-sfc-loader 能派上用場的地方。

本文介紹 vue3-sfc-loader 的運作方式、配置選項，以及幾個實際應用場景。

什麼是 vue3-sfc-loader？

vue3-sfc-loader 讓瀏覽器可以直接載入並使用 Vue 3 的 .vue 檔案，不需要預先編譯。適合以下幾種情境：

  1. 執行期動態載入元件
  2. 線上程式碼編輯器
  3. 不使用建置工具的小型專案
  4. 原型設計與快速驗證

深入理解配置選項

vue3-sfc-loader 的配置物件主要有以下選項：

  1. moduleCache：快取已載入的模組，避免重複請求。vue: Vue 這行就是把 Vue 本身預先注入快取，讓 SFC 的 import { ref } from 'vue' 能解析到同一個 Vue 實例。
  2. getFile：非同步函式，負責取得 .vue 檔案內容。v0.8+ 回傳 { getContentData: asBinary => ... } 格式；舊版（v0.7 以前）可直接回傳字串。
  3. addStyle：把元件的 <style> 區塊注入 DOM 的函式。
  4. pathResolve：解析相對 import 路徑，簽名為 ({ refPath, relPath }, options) => string。SFC 內部若有 import './子元件.vue' 這類相對路徑，loader 靠這個選項決定實際要去哪裡取檔。沒正確設定時，相對 import 會以 undefined 為基底解析而靜默失敗，是實務上最常踩的坑。
  5. getResource：載入非 .vue 資源時被呼叫，簽名為 (pathContext, options) => Resource。回傳 undefined 表示交由 loader 預設行為處理。SFC 需要引入 CSS、JSON 等資源時，可透過此選項攔截並自訂載入邏輯。
  6. handleModule：攔截特定模組類型（如 .css、.json），也可用來載入有 CJS/UMD 版本的第三方函式庫。回傳 undefined 則退回預設處理。
  7. log：自訂日誌輸出的函式。

注意：URL.revokeObjectURL(url) 在 loadModule resolve 後立即釋放 Blob URL。對自包含（無相對 import、無 <style scoped>）的單一 SFC 沒問題；若 loader 內部需要二次讀取同一 URL（如處理 scoped style 的 hash），URL 已失效會靜默失敗。保守做法是不釋放，或改在元件 onUnmounted 時才釋放。

另外，若使用者撰寫的元件內有 import './子元件.vue' 之類的相對路徑，loader 會以 blob: URL 為基底解析，導致找不到檔案。實際的線上編輯器（如 Vue Playground）需要同時覆寫 pathResolve 與 getFile，把相對路徑對應到編輯器的虛擬檔案系統，才能支援多檔案組合。

效能考量

在執行期編譯 SFC 會有明顯開銷，幾個關鍵數字值得注意：loader 本身 gzip 後約 500KB+，比一般 UI 元件庫還重；首次載入時瀏覽器還需要在主執行緒執行 Vue 模板編譯，對首屏效能的影響不可忽視。

實際使用時：

  • **善用 moduleCache**：把已載入的模組存入快取，避免同一個 .vue 被重複 fetch 和編譯。moduleCache 物件可以在多個 loadModule 呼叫間共用。
  • 按需載入：搭配 defineAsyncComponent 讓元件在真正需要時才觸發載入，不要在頁面初始化時一次載入全部。
  • 生產環境評估：如果應用規模擴大，考慮改回 Vite/Webpack 建置流程，讓編譯工作移到建置期而非使用者瀏覽器。

結論

vue3-sfc-loader 適合原型、插件系統、線上編輯器等「必須在執行期動態載入 .vue」的特定場景。多數生產應用仍應以建置時預編譯為主。版本方面，目前穩定版是 0.9.x，v1.x 的 API（尤其是 getResource 簽名）仍有 breaking change，引用前確認版本號。
