.NET Core 中的中間件（Middleware）詳解

什麼是中間件？

在 .NET Core 應用程式中，中間件是一系列組件，它們形成了一個請求處理管道。每個中間件都可以:

  1. 處理傳入的 HTTP 請求
  2. 處理傳出的 HTTP 回應
  3. 呼叫管道中的下一個中間件
  4. 在管道中的下一個中間件完成處理後執行某些操作

中間件的概念類似於一條組裝線，每個工人（中間件）都對產品（請求/回應）進行某些操作。

中間件的工作原理

中間件在 .NET Core 應用程式中按照特定的順序執行。這個順序很重要，因為它決定了哪個中間件先處理請求，以及回應如何被處理。

工作流程如下：

  1. 當應用程式接收到 HTTP 請求時，請求進入中間件管道。
  2. 每個中間件組件可以:
    • 執行一些操作
    • 將請求傳遞給下一個中間件
    • 或者短路請求（不再傳遞給後續中間件）
  3. 一旦請求到達管道的末端，它會開始返回。
  4. 回應會再次通過所有中間件，讓每個中間件有機會處理或修改回應。

常見的中間件

.NET Core 提供了許多內建的中間件，以下是一些常見的例子：

  1. 認證中間件: 處理使用者認證，確保只有授權的使用者可以存取某些資源。

  2. CORS 中間件: 處理跨來源資源共享（CORS），允許或限制來自不同網域的請求。

  3. 靜態檔案中間件: 處理靜態檔案的請求，如 HTML、CSS、JavaScript 和圖片等。

  4. 路由中間件: 根據 URL 將請求導向正確的控制器和動作。

  5. 例外處理中間件: 捕獲並處理應用程式中發生的例外。

  6. 回應壓縮中間件: 壓縮回應以減少傳輸的資料量，提升效能。

自訂中間件

除了使用內建的中間件，開發者還可以建立自訂中間件來滿足特定需求。自訂中間件通常包含：

  1. 一個公開建構函式，帶有一個型別為 RequestDelegate 的參數。
  2. 一個名為 Invoke 或 InvokeAsync 的公開方法，回傳一個 Task。

結論

中間件的順序直接影響應用程式行為——認證必須在授權之前、路由必須在端點對應之前，這些都不能隨意調換。內建中間件已覆蓋大多數場景；只有內建不夠用時，再考慮自訂。
